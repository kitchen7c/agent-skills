# Brent 72H Derivatives Report

## Overview

把用户的高约束提示转成一份可复用的布伦特原油 72 小时衍生品战术研究工作流。核心目标不是泛泛点评油价，而是生成一份可执行、可审计、可复核的短周期期货与期权联合策略报告。

这类任务的关键不是“写得像报告”，而是先做模式判定、数据分层、定量一致性校验，再输出固定结构的中文成稿。任何公开源拿不到的数据，都必须明示缺口，而不是用看起来专业的字段补空。

## 何时使用

- 用户要求 24-96 小时 Brent / ICE Brent / 布伦特原油的战术判断
- 用户要求明确方向、明确价格目标、分层区间、期货与期权搭配
- 用户要求 Black-76 或 BSM 代理建模
- 用户要求 Greeks、VaR、CVaR、压力测试、HV/IV/VRP
- 用户要求严格区分“可观测数据”“代理指标”“推断”
- 用户要求最终成稿必须是简体中文、机构研究风格、固定章节

## 不适用场景

- 长周期年度油价展望
- 纯宏观基本面评论、完全不需要衍生品结构
- 没有任何数据抓取空间且用户也不接受仅逻辑版结论
- 用户要的是实时交易执行建议，但你拿不到实时链路与盘口

## 核心原则

1. 先判模式，后做分析。
2. 先抓公开数据，后给方向判断。
3. 同一份报告里的定价、Greeks、模拟、压力测试必须使用同一套标的、到期、年化与乘数假设。
4. 不能把“推断”写成“观测”。
5. 不能把“代理指标”写成“原生指标”。
6. 不能把没有拿到的链路数据伪装成已抓取的期权链。

## 关键输入契约

先把这份任务当作“有输入门槛的研究任务”，不是“只要格式完整就能生成”的写作任务。只要关键输入缺失，就必须降级，不能为了凑齐章节去补数字。

量化输出的最小输入集如下：

- 主参考标的与对应时间戳
- 明确的到期日或持有期假设
- 可验证的波动率输入：IV、IV proxy 或 HV
- 明确的合约乘数或单位换算
- 至少一个可复核的价格路径假设或分布假设

头寸级风险输出还额外要求：

- 用户明确给出当前头寸或待建仓结构
- 头寸方向、规模、执行价、到期、乘数完整可复核
- 若为组合头寸，需给出每条腿的 side / quantity / strike / expiry

如果以下任一项缺失：

- Brent M+2 / M+3 / M+4 无法完成双源交叉验证
- 无法确认使用 Black-76 还是 BSM proxy
- 无法取得期权链或可验证的 IV / IV proxy
- 无法实际运行 Python 定量计算

则必须执行以下降级规则：

1. 明确写 `证据受限下的研究判断`
2. 保留报告结构，但量化部分只允许写方法、缺口和定性含义
3. 禁止输出任何具体的 Greeks、99% VaR、99% CVaR、Monte Carlo 概率、理论定价结果
4. 禁止把模型占位数、经验常数或常识级猜测写成计算结果

只有当最小输入集齐全且计算确实已运行时，才允许输出具体量化数字。

如果用户没有提供头寸，则所有 VaR / CVaR 表述必须限制为以下两类之一，并显式标注：

- `市场级风险度量`：描述标的或期限结构本身的价格分布与尾部波动
- `单位头寸风险度量`：描述“每手 / 每桶 / 每个模型化策略”的归一化损益风险

在无用户头寸时，禁止使用以下表述：

- `你的 VaR`
- `组合 VaR`
- `账户 VaR`
- `当前头寸 VaR`

也不得把单位头寸 VaR 冒充为用户真实持仓损失分布。

## 输出语言与成稿红线

- 最终报告必须全部使用简体中文。
- 不要出现寒暄、聊天式过渡、免责声明套话。
- 不要声称跑过 Python、Monte Carlo、链路分析，除非实际已执行。
- 不要伪造精确时间戳、期权链字段、GEX、dealer positioning、gamma wall、OI 簇、COT 数值。
- 如果公开源不可得，必须逐项写 `公开源不可得`。
- 如果使用替代指标，必须逐项写 `代理指标`。
- 如果结论来自推理而非直接观测，必须逐项写 `推断`。
- 所有日期时间都使用绝对日期与时间，不用“今天”“最新”“昨晚”。

## 执行流程

### 1. 模式闸门

先判断以下两种模式中的哪一种成立：

#### Mode A: Black-76 执行模式

仅在“当前公共可访问的 Brent futures options chain”满足以下条件时使用：

- 可见 expiry
- 可见 strike
- 可见 option side
- 且至少可见 option price 或 implied volatility

如果满足：

- 以 Black-76 为主模型
- 所有期权定价与 Greeks 统一锚定到同一 Brent futures expiry
- 行权价优先使用公开可见链路中的真实 strike

#### Mode B: BSM 代理模式

如果无法拿到完整、当前、可用于执行层面分析的 Brent 期货期权链，则必须切到 Mode B。

如果切到 Mode B：

- 使用 BZ=F 或其他可验证的 Brent 现货/结算代理
- 用 BSM 做理论定价
- 明确写明这是代理模型研究报告，不是实时链路执行报告
- 所有行权价都必须标注：`模型化执行位，不代表实时可成交盘口`

### 2. 强制数据抓取

在形成预测前，必须主动检索并交叉验证：

- Brent M+2、M+3、M+4 最新价格
- 每个价格至少来自 2 个公共来源
- 精确时间戳、来源名、是 live / last close / settled proxy
- OVX 最新值与时间戳
- M+2/M+3/M+4 曲线形态与价差
- Brent 投机/资管仓位代理；若拿不到原生 Brent，则用最接近且可信的定位代理，并标注 `代理指标`
- 接下来 72 小时的跳跃催化剂：OPEC / EIA / IEA / 库存 / 制裁 / 地缘冲突 / 宏观数据 / 展期 / 节假日 / 周末流动性

交叉验证规则：

- 同一合约两个公共价格源若偏差超过 0.30%，必须同时报告两个值，并解释最终采用哪一个以及原因。
- 若市场休市，必须明确写明使用的是最近收盘价或最近结算代理。

在正式写报告前，必须先输出或先在内部完成一个 `Input Audit`，至少列出：

- 主参考标的
- 模式判定结果
- M+2 / M+3 / M+4 双源是否完成
- IV / IV proxy 来源
- HV 来源与窗口
- 到期日或持有期
- 合约乘数
- Python 是否实际运行成功

若 `Input Audit` 有任一项为空，直接触发降级，不得继续产出伪精确量化结果。

### 3. Python 定量引擎

如果 Python 可用，必须用它做确定性计算。不要在最终报告中输出原始代码。

如果 Python 不可用：

- 明示 `量化引擎不可用`
- 不要伪造 Monte Carlo、Greeks、VaR、CVaR
- 报告降级为“数据与逻辑版”

如果 Python 可用：

- 使用 ACT/365
- HV 优先用 20 日日对数收益率；没有则退化到 10 日
- Monte Carlo 路径数不少于 100,000
- 量化输出必须同时支持期权结构与期货情景阶梯

只有在“Python 实际运行成功 + 输入审计通过”同时满足时，才允许：

- 输出具体 Greeks 数值
- 输出具体 99% VaR / 99% CVaR
- 输出基于模拟的 HDI 概率与尾部区间概率
- 输出理论期权价格与净权利金

### 4. 波动率框架

必须计算或估算：

- HV
- IV 或 IV proxy
- VRP = IV - HV

解释规则：

- 如果 IV 明显高于 HV，只有在事件风险不极端时，才可以考虑定义风险的卖波动结构
- 即使 IV 显著高于 HV，也不能把短波动写成“安全”
- 如果事件跳空风险主导，优先有限损失结构

### 5. 微观结构与交易逻辑

必须从以下维度交叉验证：

- 期限结构
- skew 或 skew proxy
- positioning
- IV vs HV
- event risk

不要提及以下概念，除非真的拿到了可信公共源：

- Gamma Wall
- dealer positioning
- GEX
- OI magnet

若无法验证，直接写 `公开源不可得`。此时只能基于以下内容做 `推断`：

- 公开 strike 聚集
- 整数关口行为
- 曲线形态
- 事件结构

### 6. 区间体系

必须使用同一模型输入集，输出分层区间，而不是只给一个范围：

- 1 个波动率感知的窄战术区间
- 至少 1 个 HDI 区间，建议给 50% / 68% / 80%
- 68% 置信区间
- 95% 置信区间

区间命名规则：

- 如果窄区间概率 < 15%，不要叫“高概率区间”
- 这时只能称作 `最高密度窄区间` 或 `模式区间`
- 如果使用固定宽度窄区间，必须解释该宽度为什么在当前价格与波动 regime 下有决策意义

必须显式说明：原油分布存在非对称肥尾，上行地缘供给冲击与下行需求走弱并不对称，因此正态扩散假设对尾部可靠性有限。

### 7. 策略选择

必须给出：

- 1 个首选期货策略
- 1 个首选期权结构
- 1 个期货-期权配对框架

策略选择规则：

- 若 M+2 / M+3 / M+4 存在显著 backwardation 或 contango，必须评估 calendar spread / curve trade 是否更合适
- 若 IV 相对 HV 偏贵且事件风险可控，可考虑定义风险卖波动
- 若跳跃风险占优，应优先有限风险结构
- 不要推荐裸卖无限风险结构

期货部分必须明确：

- 使用的具体合约或跨期价差
- 入场区
- 失效位
- 目标位
- 每张合约关键价格下的盈亏阶梯

期权部分必须明确：

- 精确腿结构
- strike
- net premium
- break-even
- max profit
- max loss
- estimated probability of profit

如果用户没有提供真实持仓，所有策略风险指标默认只能对应：

- 示例结构
- 模型化建仓方案
- 单位头寸

不得暗示这些风险数字已经映射到用户现有账户。

Mode B 下必须追加一句：

`以上执行价为模型化执行位，不代表实时可成交盘口`

### 8. 一致性自检

定稿前必须检查：

- 定价、Greeks、模拟、压力测试是否使用同一标的参考
- 到期日与 day count 是否一致
- 合约乘数是否一致
- 有界风险结构的 max loss 是否与结构几何一致
- 99% VaR / 99% CVaR 绝对值不能超过理论 max loss
- 压力测试损益不能与 max loss 冲突
- 期货 PnL ladder 是否与合约乘数一致
- 窄区间措辞是否与真实概率一致
- 是否误把未验证的 gamma / dealer / OI 结论写成事实

如果任何一项不通过，先修正，再输出。不要把草稿错误直接留给用户。

## 每一节的最低内容要求

### 1. 战术执行摘要

- 明确方向：`偏多` / `中性偏震荡` / `偏空`
- 明确窄目标区间
- 明确该区间是真高概率区间还是仅为最高密度窄区间
- 明确首选期货策略
- 明确首选期权结构
- 明确模式：`Black-76 执行模式` 或 `BSM 代理模式`

### 2. 底层数据与时间戳

- 写清 Brent M+2 / M+3 / M+4 的采用价格、时间戳、来源、数据类型
- 写清 OVX 与时间戳
- 写清曲线形态
- 写清仓位代理
- 写清 72 小时跳跃催化剂
- 写清为何选 Mode A 或 Mode B

### 3. 交叉验证

- HV
- IV / IV proxy
- VRP
- 期权贵/平/便宜的判断
- 市场是 panic-priced / fairly priced / complacent 的哪一种
- 各信号一致还是背离

### 4. 72 小时价格预测

- 窄战术区间
- 该区间真实概率
- 至少一个 HDI
- 68% 区间
- 95% 区间
- 明确窄区间属于高概率区间还是仅为最高密度窄区间

### 5. 联合策略

- 期货策略与期权结构都要展开
- 说明何时期货优先、何时期权优先
- 说明二者如何配对，而不是重复暴露

### 6. 情景分析

至少选 2 个当前市场真正相关的关键情景，不要机械使用 110 / 120 这类固定点位。每个情景必须说明：

- 为什么该价位重要
- 对市场 regime 的含义
- 期货仓位如何表现
- 期权结构如何表现
- 应该持有 / 减仓 / 平仓 / 展期
- 如果要替换结构，替换成什么

### 7. 逐日交易指令

必须给出明确数字化规则：

- 入场条件
- 止盈条件
- 止损条件
- gap risk 应对
- IV crush 应对
- 交易 thesis 失效条件

### 8. 量化引擎输出

仅在“输入审计通过且 Python 实际运行成功”时，才必须包含以下表格或结果：

- 每个期权腿的 markdown 表：leg / strike / theoretical price / assumed IV / side / net premium
- 组合净 Greeks 表：Delta / Gamma / Theta per day / Vega per 1 vol point
- 首选期货策略 PnL ladder 表
- 99% VaR
- 99% CVaR
- `+15%` 地缘肥尾冲击压力测试
- `+10%` 压力测试
- `-10%` 压力测试

VaR / CVaR 输出时必须先标明口径：

- `市场级风险度量`
- `单位头寸风险度量`
- `头寸级风险度量`

其中 `头寸级风险度量` 仅在用户明确提供头寸后允许出现。

如果未通过输入审计，`【📊 量化引擎运算与一致性复核结果 | Quantitative Engine Output】` 一节必须改为以下内容：

- `状态：降级`
- 缺失的关键输入清单
- 原本需要但当前禁止输出的量化项目清单
- 这些缺口对策略可执行性的影响
- 下一步需要补齐的数据与工具

如果输入审计通过但用户未提供头寸，也必须额外写明：

- 当前仅允许输出市场级或单位头寸风险度量
- 当前禁止输出账户级、组合级、真实持仓级 VaR / CVaR

### 9. 术语附录

使用 markdown 表，列：

- term
- definition
- role in this report

只定义本报告中真的出现过的术语。若术语是代理意义下使用，定义中必须说明。

## 常见错误

- 只给单一区间，不给分层区间
- 方向判断含糊，没有明确偏多/偏空/震荡
- 期货与期权策略重复承担同一风险，而不是做配对
- 用 spot 做了一个模型，却把结果写成 futures execution note
- 给出 Greeks，但前后 underlying / expiry / multiplier 不一致
- 说用了实时期权链，但实际上并没有可验证链路
- 拿 OVX 直接等同于 Brent 原生波动率，而没有标注 `代理指标`
- 把没有公开来源支持的微观结构叙述写成事实
- 没有完成 `Input Audit`，却硬写 VaR、CVaR、Greeks、theoretical price
- 因为输出模板要求完整，就用占位假设伪造量化结果
- 用户未提供头寸，却把单位头寸 VaR 写成账户或组合 VaR

- 定量引擎结果已生成，或已明确降级
- 风险指标与结构几何一致
- 若输出 `头寸级风险度量`，用户头寸已完整提供
- 全文为简体中文
- 固定章节完整
- Sources 已列出

否则，明确写明限制，并降级为“证据受限下的研究判断”。
